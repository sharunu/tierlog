## 学び / 振り返り

### codex レビューフローの効果
plan→codex 往復で 8 点の指摘がすべて反映され、最終レビューでも P2 1 件のみ。事前検証が機能した。

### Supabase Storage の upsert と SELECT policy
- public bucket でも upsert は SELECT 権限を要求 (公式 docs)
- `upsert: false` (本 plan) で INSERT のみ実行 → SELECT policy 不要 (将来削除可、Phase E2)
- 既存 SELECT policy は残置 (regression 保険、本人 prefix 限定で listing 漏洩なし)

### handle_new_user の signup trigger 設計
- `auth.users` トリガなのでアプリ上の選択ゲームを知らない
- → 全新規非ゲストユーザーに dm + pokepoke 両方の major デッキが入る挙動を受け入れた
- 表示は UI 側の `game_title` フィルタで分岐するため混入なし
- ポケポケ name の英語問題は別 fix で解決 (`COALESCE(name_ja, name)`)

### Plan-critic と codex の使い分け
- plan-critic: 機械的整合性 (ファイル存在、行番号、SQL 構文、既存パターン一致) を網羅検証
- codex: 設計判断・暗黙挙動・運用観点 (REVOKE 明文化、catch 対象分離、try/finally 等) を補完
- 両者を併用すると重複指摘が少なく、一周で精度高く plan が固まる
