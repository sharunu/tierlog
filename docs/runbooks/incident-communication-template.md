# runbook: ユーザー周知文テンプレート

最終更新: 2026-05-25

## いつ参照する

- 本番障害でユーザー側に影響が出ている、または出た可能性がある時
- 5 分以上の影響時間 または ユーザー報告ありで、周知判断した時
- 計画的なメンテナンス（DB マイグレーション等）を実施する時

## 周知チャネルの優先度（個人開発スコープ）

1. **X（旧 Twitter）公式アカウント**: tierlog 運営アカウント。最も到達範囲が広い
2. **Discord 関連サーバー**: tierlog 連携している Discord サーバー（コミュニティがある場合）
3. **tierlog.app 内お知らせバナー**: 実装次第。緊急性が高い場合は GitHub Pages or 静的 HTML を Cloudflare Workers で配信する手も

## テンプレート集

### A. 障害発生時（速報）

```
【障害発生のお知らせ】

現在、tierlog で <現象を簡潔に> の問題が発生しています。
原因を調査中です。ご不便をおかけし申し訳ありません。

復旧次第、改めてお知らせします。
お問い合わせ: contact@tierlog.app
```

ガイドライン:
- 30 分以内に状況更新するか、復旧次第通知
- 「データが消えた」等のセンシティブな表現は避け、調査中であることを伝える
- 連絡窓口（contact@tierlog.app）を必ず併記

### B. 障害復旧時

```
【復旧のお知らせ】

先ほどお知らせした tierlog の <現象> は解消しました。

ご利用中のデータへの影響: <なし / 一部復旧 / 詳細調査中>
発生時間: 2026-MM-DD HH:MM 〜 HH:MM (合計 X 分)

ご不便をおかけし申し訳ありませんでした。
詳細・お問い合わせ: contact@tierlog.app
```

ガイドライン:
- 「データへの影響」を明示。不明な場合は「詳細調査中」「個別にお問い合わせください」と書く
- 発生時間を分単位で。長時間障害の場合は理由も簡潔に

### C. 計画メンテナンス（事前告知）

```
【メンテナンスのお知らせ】

下記の時間帯、tierlog でメンテナンスを実施します。

日時: 2026-MM-DD HH:MM 〜 HH:MM (JST、約 X 分間)
影響: <一時的にアクセスできなくなります / ログイン状態が解除されることがあります 等>
目的: <DB スキーマ更新 / セキュリティ更新 等、簡潔に>

メンテナンス完了次第、改めてお知らせします。
ご不便をおかけしますが、よろしくお願いします。
```

ガイドライン:
- 少なくとも 24 時間前には告知
- 影響内容を具体的に（「ログインできない」等）

### D. データ事故告知（最大限慎重）

```
【重要なお知らせ：データの取り扱いについて】

2026-MM-DD 頃に発生した tierlog のシステム上の問題により、<事象> が発生したことが判明しました。

影響を受けた可能性のあるユーザー数: <概数>
影響を受けた可能性のあるデータ: <戦績の一部 / 設定情報 等>
発生原因: <簡潔に>
現在の対応状況: <調査中 / 復旧済 / 個別対応中 等>

該当する可能性のある方は、お手数ですが contact@tierlog.app までお問い合わせください。

ご利用の皆様には多大なご不便をおかけし、心よりお詫び申し上げます。
```

ガイドライン:
- 個人情報保護委員会のガイドラインに基づき、必要な情報を遅延なく開示
- 専門家確認（弁護士等）を経るのが望ましいが、緊急性に応じて速報→詳細の二段階で出す
- 連絡窓口を必ず明示

## 周知後の対応

- 発生から復旧までの経緯を `docs/reports/YYYY-MM-DD_<incident-name>.md` で記録
- 同種事故の再発防止策を `docs/runbooks/` の関連 runbook に反映

## 公式参照（ベストプラクティス）

- Cloudflare incident communication best practice（参考）: https://blog.cloudflare.com/transparent-product-development/ （取得日 2026-05-25）
