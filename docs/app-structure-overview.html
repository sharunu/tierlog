1. このアプリは何？

Tierlogは、デジタルカードゲームの対戦記録を保存し、勝率やデッキの相性、環境（よく当たる相手デッキ）を可視化する Web アプリです。元々はデュエル・マスターズ プレイス（dm）専用でしたが、現在はマルチゲーム対応で、ポケポケ（pokepoke）も同じアプリ内で扱えるようになっています。

ユーザーができる主なこと
  • 対戦結果を 1 タップで記録（自分のデッキ／相手のデッキ／勝敗／メモ）
  • 勝率・対面相性・環境分布をグラフで確認
  • Discord 連携で「チーム」を作り、メンバー同士で戦績を共有
  • 戦績ページを画像化してSNSにシェア（OGP動的生成）
  • 運営による不正検知・品質スコアリングで荒らしデータを除外

2. 全体アーキテクチャ

このアプリは「ブラウザ ←→ Cloudflare Workers ←→ Supabase」という3層構造です。サーバーサイドの処理はほぼ無く、ブラウザから直接 Supabase に話しかけるシンプルな構成です。

① ユーザーの端末（ブラウザ / PWA）

React で書かれた画面が動く場所。ユーザーの操作を受け取り、データを Supabase に問い合わせる。

Next.js (App Router)
画面の組み立てとルーティング
React 19
UI部品の表示・更新
supabase-js
DB通信ライブラリ
PWA Service Worker
アプリのようにインストール可能
▼ HTTPS

② Cloudflare Workers（配信レイヤー）

アプリの HTML/JS/CSS を世界中の CDN から高速配信。一部のサーバー処理（OGP画像生成、Discord OAuth、内部APIなど）もここで動く。

OpenNext for Cloudflare
Next.js を Workers で動かすアダプタ
Workers Rate Limiting

▼ Supabase JS Client (HTTPS)

③ Supabase（バックエンド一式）

DB・認証・ストレージをまとめて提供するサービス。アプリのデータはほぼ全てここに保存される。

PostgreSQL DB
対戦・デッキ・チーム等のデータ本体
Supabase Auth
X / Google / メールログイン
Row Level Security
行単位の権限制御（重要！）
Storage
共有用の画像保存
ポイント ─ サーバー側で動くコードは最小限。ブラウザから直接 Supabase を叩く設計で、その代わりRow Level Security（RLS）という DB レベルの権限制御で安全性を担保しています。

7. データの流れ

ユーザーが「対戦結果を記録」ボタンを押した時、データはどう流れるか？

1ユーザー操作
ブラウザで「保存」ボタンをタップ
2React コンポーネント
BattleRecordForm.tsx が onSubmit を発火
3actions 層
battle-actions.ts の recordBattle() を呼ぶ
4Supabase クライアント
supabase.from('battles').insert(...) 実行
5RLS チェック
DB 側で「このユーザーは自分の行しか書けない」を検証
6DB 書き込み
PostgreSQL に保存。トリガーで派生処理（相手デッキ自動追加等）

src/lib/actions/ の中身

ファイル行数担当
account-actions.ts166アカウント情報の取得・退会・X連携
admin-actions.ts849管理者機能（ユーザー一覧・BAN・相手デッキマスタ管理）
battle-actions.ts286対戦の記録・編集・削除・履歴取得
deck-actions.ts169自分のデッキの作成・編集・並び替え
stats-actions.ts958勝率・期間集計・環境分布の取得（一番大きい）
team-actions.ts192Discord 連携・チーム表示切替・メンバー取得
feedback-actions.ts16ユーザーフィードバック送信

actions 層について ─ ファイル名に "actions" とありますが、Next.js の Server Actions は使っていません。これらはクライアントから呼ぶ普通の async 関数で、中で supabase-js を呼んでいるだけです。「画面とDBの間に挟む薄い層」と理解すればOK。

9. ログインの仕組み

ログインは Supabase Auth の implicit flow を使っています。流れは次の通り。

1/auth 画面
X / Google / メールリンク のボタンを押す
2外部プロバイダ
X や Google のログイン画面にリダイレクト
3/auth/callback
URL のハッシュからトークン取得 → セッション確立
4middleware.ts
毎リクエストで Cookie からセッション復元
5ホーム画面へ
/dm/home または /pokepoke/home

関連ファイル

src/lib/supabase/client.tsブラウザ用クライアント（implicit flow 設定）
src/lib/supabase/server.tsmiddleware / API ルート用クライアント
src/middleware.tsセッション更新 + Rate limit + 旧URL転送
src/app/auth/callback/page.tsxOAuth コールバック処理
src/components/providers/BanGuard.tsxBAN ユーザーの画面遮断
変更厳禁 ─ CLAUDE.md に「既存のauth設定（implicit flow、X/Googleログイン、middleware.ts、client.ts）は変更しないこと」と明記されています。auth/callback も既存の SIGNED_IN 処理は触らず、追加分のみOK。

10. マルチゲーム対応

このアプリは元々デュエプレ専用でしたが、2026年4月のマイグレーションで 1リポジトリで複数ゲームを扱う構造に拡張されました。

ゲームレジストリ（src/lib/games/index.ts）

export const GAMES = {
  dm: {
    slug: "dm",
    trackerName: "Tierlog - デュエプレ",
    formats: [{ code: "ND" }, { code: "AD" }],
    defaultFormat: "ND",
  },
  pokepoke: {
    slug: "pokepoke",
    trackerName: "ポケポケトラッカー",
    formats: [{ code: "RANKED" }, { code: "RANDOM" }],
    defaultFormat: "RANKED",
  },
};

URL 構造のルール

パターン例説明
/{game}/.../dm/home, /pokepoke/battleゲーム別ページ
/account, /auth, /admin—ゲーム非依存（共通）
/share/[id]—シェア URL（中で game_title 解決）

DB のゲーム分離

ゲーム別のテーブル（decks, battles, deck_tunings, opponent_deck_master, shares, teams, discord_connections, detection_alerts）には game_title カラム があり、これでゲームをスコープ分離しています。

新ゲーム追加時の罠 ─ format コードを既存ゲームと絶対に重複させないこと。読み込み系 RPC は p_format フィルタだけで分離している前提なので、重複すると別ゲームのデータが混ざる。例: 遊戯王マスターデュエルを追加するなら RANKED_YGO のような名前にする。

11. 開発・デプロイの流れ

このプロジェクトの厳格なルール: 全ての変更は必ず dev branch 経由。main へ直接 push は禁止。

2dev に push

同じく3〜5分で反映

コマンド用途

禁止コマンド ─ npx next build / npm run deploy は通常実行不要（本番は Cloudflare 自動ビルド）。DB マイグレーション (npx supabase db push) はユーザーの明示指示後のみ、しかもmain 本番反映が完了してから実行する（dev/prod 共通DBのため）。

13. 用語ミニ辞典

App RouterNext.js の新しいルーティング方式。src/app/ 配下のフォルダ構造がそのまま URL になる。
Cloudflare Workers世界中の CDN エッジで JS を実行するサーバーレス基盤。本番アプリの実行先。
OpenNextNext.js を Cloudflare / AWS Lambda 等で動かすためのアダプタ。@opennextjs/cloudflare を使用。
SupabasePostgreSQL + 認証 + ストレージをまとめて提供する SaaS。Firebase の OSS版に近い。
RLSRow Level Security。PostgreSQL の行単位アクセス制御。「自分の行しか読めない／書けない」をDBで強制。
RPCPostgreSQL の関数。Supabase クライアントから supabase.rpc('関数名', {引数}) で呼べる。複雑な集計に使う。
SECURITY DEFINERPostgreSQL の関数属性。関数を「定義者の権限で」実行する。RLS をすり抜けられるので search_path 固定など追加対策が必要。
implicit flowOAuth の認可フローの一種。URL ハッシュにトークンを載せる方式。Supabase の SPA デフォルト。
PWAProgressive Web App。Web アプリをスマホにインストールしてネイティブアプリ風に使える仕組み。
OGPOpen Graph Protocol。SNS でURL を貼った時に表示されるサムネ画像・タイトル・説明を制御する meta タグ。
Service Workerブラウザがバックグラウンドで動かす JS。オフライン対応や push 通知の核。
middlewareNext.js で全リクエストの前に挟む処理。このアプリでは rate limit と セッション更新と旧URL転送に使用。
migrationDB のスキーマ変更を時系列で管理する SQL ファイル群。supabase/migrations/ に蓄積。
OAuth コールバック外部プロバイダ（X / Google）でログイン後に、自アプリの指定 URL に戻ってくる処理。
dev / staging / productiondev = 開発用、staging = 検証用、production = 本番用。このアプリでは dev preview が staging DB を参照する。
