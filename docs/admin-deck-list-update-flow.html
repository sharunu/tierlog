ADMIN GUIDE

対面デッキリストのDB更新フロー

この資料は、Tierlog の「対面デッキリスト」がどこから更新され、どの条件で 有効/無効、major/minor/other、追加、削除が起きるかを管理者向けに整理したものです。 対象DBは主に opponent_deck_master と opponent_deck_settings です。

全体像

Mode 1

完全管理者依存

管理者が /admin/opponent-decks でカテゴリ、表示順、有効状態、追加、削除を直接管理します。 ユーザーが未登録の対面名を入力した場合は、DBトリガーで other・有効の候補として自動追加されます。

major 手動 minor 手動 other 手動 有効 手動

Mode 2

ユーザー入力依存

ユーザーの対戦記録 battles を集計し、閾値方式またはデッキ数固定方式でカテゴリと表示順を再計算します。 未登録デッキは対戦記録のINSERT後に自動追加され、このモードでは有効になります。

major 閾値/固定件数 minor 閾値/固定件数 other 自動分類 古い候補は無効化

Mode 3

LimitlessTCG依存

ポケポケのみ。LimitlessTCG のデッキ一覧を取り込み、RANKED と RANDOM に同じスナップショットを適用します。 現状はAPIキー待ちのためHTML取得が停止され、既存キャッシュ表示のみです。

major share/順位 minor share/順位 今回取得あり 今回取得なし
重要: 管理画面では多くの操作が画面上の一時状態に積まれます。 DBに反映されるのは、基本的に右上の 「変更内容反映」 を押したタイミングです。 ただし Limitless の和名編集だけは、入力欄のフォーカスアウト時に専用RPCで即時保存されます。
対面デッキリストの自動追加・最終使用更新の対象は battles.opponent_deck_name です。 使用デッキ側の my_deck_id / my_deck_name は decks テーブルで別管理され、対面デッキリストの追加トリガーには使われません。

DBの役割

opponent_deck_master

対面デッキリスト本体です。ユーザー画面の候補、統計画面の major/minor/other 集約、管理画面の一覧がここを参照します。

nameアプリ内の正式名。現在は空白なしに正規化されます。
game_title / formatゲームとフォーマットのスコープ。例: dm/ND, pokepoke/RANKED
is_activeユーザー候補に出すかどうか。無効でも過去戦績は消えません。
categorymajor, minor, other。候補表示と統計の集約に使います。
sort_order候補表示順。管理者手動、autoの自動計算、Limitless share順で更新されます。
last_used_atユーザーが対戦記録に入力した最終時刻。autoモードの無効化判定に使います。
admin_bonus_countautoモードの分類計算で、管理者が任意に加算できる仮想対戦数です。
sourcemanual または limitless が実運用上の主な値です。未登録入力からの自動追加も現状はデフォルトで manual になります。

opponent_deck_settings

ゲーム/フォーマットごとの管理方式と自動分類ルールを持ちます。

management_modeadmin, auto, limitless
major_thresholdauto/Limitlessの閾値方式で major になる使用率またはshare。
minor_thresholdauto/Limitlessの閾値方式で minor になる使用率またはshare。
usage_period_daysautoモードで集計対象にする直近日数。
disable_period_daysautoモードで未使用候補を無効化する日数。
classification_methodauto/Limitlessで使う分類方式。threshold または fixed_count。
major_fixed_count / minor_fixed_count固定件数方式で、上位何件を major/minor にするか。autoでは使用数、Limitlessではshare順位を使います。

カテゴリ/順序/無効化

閾値方式または固定件数方式で major/minor/other、古い候補は無効化。

分類方式の考え方

閾値方式 threshold

集計対象は usage_period_days 日以内の battles です。 分母は 期間内の対戦数 + 有効デッキの admin_bonus_count 合計。 各デッキの使用率は (そのデッキの対戦数 + admin_bonus_count) / 分母 * 100 で計算されます。

使用率 >= major_threshold 使用率 >= minor_threshold それ未満

デッキ数固定方式 fixed_count

有効デッキを 対戦数 + admin_bonus_count の使用数降順で並べ、 上位 major_fixed_count 件を major、続く minor_fixed_count 件を minor にします。 使用数が0件のデッキは、固定枠が余っていても必ず other です。

上位 major_fixed_count 次の minor_fixed_count 使用数0は other 固定
固定件数方式では、古い有効デッキの無効化を先に実行してから分類します。 これにより、固定枠は再計算時点で有効なデッキだけで埋まります。
項目どう更新されるかDB更新ルート
有効化 新規対面名が入力されると有効で追加。無効だった既存デッキが再使用されると有効に戻ります。 auto_add_opponent_deck
無効化 last_used_at が disable_period_days より古い有効デッキ、または未使用で作成日が古い有効デッキを無効化します。 _recalculate_opponent_decks_internal
major/minor/other 分類方式に応じて自動更新。閾値方式は使用率、固定件数方式は使用数順位で分類します。管理画面の「試し計算」はカテゴリのプレビューのみで、DB更新や自動無効化はしません。 recalculate_opponent_decks または run_daily_opponent_deck_batch
表示順 カテゴリ順、次に 対戦数 + admin_bonus_count 降順、最後に名前順。 _recalculate_opponent_decks_internal
日次再計算はDB側の daily-opponent-deck-recalc cron から run_daily_opponent_deck_batch() を呼び、dm/AD, dm/ND, pokepoke/RANKED, pokepoke/RANDOM を順に処理します。 management_mode が auto 以外のフォーマットは内部で何もせず終了します。

LimitlessTCG依存

ポケポケのみの外部スナップショット同期 management_mode = limitless
現在のコードでは LIMITLESS_HTML_SYNC_PAUSED = true です。 そのため、管理画面の「今すぐ取得」は無効、GitHub Actions や内部APIから呼んでも skipped で終了し、新規取得・カテゴリ再計算は行われません。 以下は再開後に動く設計上のフローです。

モード切り替えトリガー

候補として表示される条件

ユーザーの対戦入力候補や統計画面のカテゴリ分けは、基本的に get_opponent_deck_suggestions(p_format) が返す is_active = true の master 行に依存します。 無効行や削除済み行は候補に出ません。ただし過去の battles は残るので、統計自体には名前が残り得ます。

更新ルート一覧

画面/APIルート

/admin/opponent-decks

管理者用の対面デッキ管理画面。モード変更、分類方式/閾値/固定件数変更、追加、削除、有効切替、カテゴリ変更、Limitless和名編集を行います。

/dm/battle / /pokepoke/battle

ユーザーの対戦記録入力画面。保存時に battles へINSERTし、DBトリガーで master の追加/last_used_at 更新が起きます。

/api/admin/limitless-sync

管理者セッションのBearer JWTで呼ぶLimitless同期API。内部でservice role clientを使います。

/api/internal/limitless-sync

GitHub Actionsなどの内部実行用。X-Internal-Key が必要です。

DB関数/トリガー

battles_auto_add_opponent_deck

battles のAFTER INSERTトリガー。auto_add_opponent_deck を呼びます。

auto_add_opponent_deck

対戦記録から master の存在確認、追加、last_used_at 更新を担当。新規追加は admin/auto が有効、Limitless が無効。既存行の再有効化は auto のみ。

recalculate_opponent_decks

管理者がautoモードの反映時に呼ぶ公開RPC。admin判定後、内部再計算へ進みます。

run_daily_opponent_deck_batch

日次cronから呼ばれるDB関数。全ゲーム/フォーマットのauto再計算を試みます。

apply_limitless_snapshot

Limitless同期の本体。スナップショットを master に反映し、カテゴリと表示順も更新します。

admin_update_opponent_deck_name_ja

Limitless由来デッキの和名編集用。master名と過去battle名を同じtransactionで同期します。
